## 버그버디 v1 PRD

### 1. 개요

- **제품 이름(가칭)**: 버그버디
- **목표**: 사업팀/운영팀이 레드마인에 등록하는 이슈를 구조화하고, 고객사·레포 컨텍스트를 활용해 **개발자가 어느 화면/소스를 먼저 봐야 할지 빨리 파악**하도록 돕는 내부용 도구.
- **핵심 가치**
  - 이슈 입력 시 필수 정보를 빠짐없이 받는다.
  - 레드마인 이슈 Description을 일정한 형식으로 정리한다.
  - 고객사·키워드·에러 정보를 기반으로 **코드 후보 파일 Top-N**을 추천해 초기 트리아지 시간을 줄인다.

### 2. 범위 (v1 스코프)

- **타겟 사용자**
  - 사업팀/운영팀: 고객 이슈를 정리해 레드마인에 등록하는 담당자.
  - 개발자: 레드마인 이슈를 보고 원인 분석을 시작하는 담당자.

- **기능 범위 (선택 결과 반영)**
  - **내부용 + 풀 기능 세트 (v1\_internal\_full)**:
    - 레드마인 이슈 템플릿 생성 (복붙용).
    - 레드마인 REST API를 통한 **이슈 조회(read-only)**:
      - Tracker/Status 목록 등 메타데이터 조회.
      - (향후) 기존 이슈 내용 개선 지원까지 고려.
    - 고객사 → (repo/branch/local\_path) 매핑에 기반한 **키워드 기반 코드 후보 추천**.

- **Out of Scope (v1)**
  - 레드마인 이슈 생성/업데이트 API 호출(쓰기): v1에서는 **읽기 전용 연동**까지만.
  - 고객이 직접 접속하는 외부 포털.
  - 고급 코드 인덱싱/시맨틱 검색(LLM이 직접 리포 전체를 인덱싱하는 수준).
  - 대규모 MLOps/로그 파이프라인 연동.

### 3. 성공 지표

- **가장 중요한 1개 (선택 결과)**:
  - **개발자가 이슈를 처음 열고 “어디를 봐야 할지” 결론을 내리기까지 걸리는 시간이 줄어든다.**
    - 정성 지표: “이슈요정 덕분에 첫 10분 헛돌던 시간이 줄었다”는 개발자 피드백.
    - 정량 후보:
      - 랜덤 샘플 이슈들에 대해, 담당 개발자가 “처음 확인한 파일/페이지가 실제 수정 영역과 동일/매우 인접”한 비율.

### 4. 핵심 시나리오 (User Stories)

1. **사업팀 이슈 등록**
   - 사업 담당자는 고객사 이슈를 접수한다.
   - 이슈요정 웹앱에 접속해:
     - 고객사를 선택한다.
     - Tracker, Status(기본 New), Start date를 선택/입력한다.
     - 에러/문의 내용을 한국어로 입력하고, 메뉴명/키값/재현절차/기대 vs 실제 결과/에러 메시지 등을 채운다.
   - “템플릿 생성” 버튼을 누르면 레드마인 Description 형식으로 정리된 내용을 얻는다.
   - 사업 담당자는 이 내용을 레드마인 이슈에 복붙하거나(기본), 이후 제공될 API 연동을 참고한다.

2. **누락 정보 체크 + 추가 질문 리스트 생성**
   - 이슈요정은 Claude(텍스트만 사용)를 통해 입력된 내용을 분석한다.
   - 누락됐을 가능성이 높은 필드(예: 재현 절차, 문서키, 환경, 발생 시각 등)를 탐지해 **추가로 고객에게 물어봐야 할 질문 리스트**를 제안한다.
   - 사업 담당자는 이 질문 리스트를 고객에게 전달하거나, 내부적으로 보완한다.

3. **개발자의 초기 트리아지**
   - 개발자는 레드마인 이슈 링크를 열고, Description에 “이슈요정 템플릿”이 붙어 있는 것을 본다.
   - 이슈요정 앱에서 같은 이슈 내용을 넣고, 고객사와 키워드(메뉴명, API 경로, 에러 메시지 일부)를 입력한다.
   - 이슈요정은 고객사에 매핑된 repo/branch/local\_path 범위에서 키워드로 파일을 검색해 **후보 파일 Top-N**을 보여준다.
   - 개발자는 이 후보 리스트를 기반으로 IDE에서 바로 해당 파일을 열어 분석을 시작한다.

### 5. 기능 요구사항

#### 5.1. 이슈 입력 & 템플릿 생성

- **입력 필드**
  - Tracker: `Common / Feature / Defect / Request / Patch / Issue / Review` 중 선택.
  - Status: `New / Confirm / Assigned / In progress / Resolved / Closed / Need Feedback / Rejected` (기본 New).
  - Start date: 날짜 선택 (기본 오늘).
  - Priority: `낮음 / 보통 / 높음 / 즉시`
  - Category: `SFE / 주문,반품,수금 /지출보고`
  - 고객사: `웹취약점 개선 / DNC / FMC / 넥스팜 / 다케다 / 대웅바이오 / 대웅제약 / 동구바이오 / 메드트로닉 / 박스터 / 벡톤디킨스코리아 / 보령컨슈머헬스케어 / 삼일제약 / 성보화학 / 시지바이오 / 신풍제약 / 종근당 / 캔논메디칼시스템즈 / 한국다이이찌산쿄 / 한국아스텔라스 / 한국코와 / 한독 / 한옥바이오 / 휴온스`
  - 메뉴명
  - URL/라우트(알면).
  - 사용자 ID.
  - 문서 키/번호.
  - 기타 키값.
  - 발생일시(알면).
  - 브라우저/앱 정보(알면).
  - (1) 에러 내용/이슈사항/문의내용 (멀티라인 텍스트).
  - (5) 재현 절차 (멀티라인 텍스트).
  - 기대 결과 / 실제 결과.
  - (7) 에러 메시지/로그(원문) (멀티라인 텍스트).

- **출력**
  - 레드마인 Description에 그대로 붙여넣을 수 있는 **한국어 템플릿**:
    - 섹션 구조:
      - (1) 에러 내용/이슈사항/문의내용
      - (2) 메뉴명 / 페이지 정보
      - (3) 상세 키값
      - (4) 발생 환경/시간
      - (5) 재현 절차
      - (6) 기대 결과 vs 실제 결과
      - (7) 에러 메시지/로그(원문)
  - 필드가 비어 있을 경우 `UNKNOWN` 또는 `입력되지 않음` 등 통일된 placeholder 사용.

#### 5.2. Claude 기반 누락 체크 & 질문 생성 (텍스트만)

- **입력**
  - 위 템플릿으로 조합된 초안 + 원문 이슈 텍스트(필요시).
  - Tracker/Status/Start date/고객사 이름.

- **처리**
  - Claude(예: `claude-3-5-sonnet-latest` 또는 `haiku`)에 **텍스트만** 전달 (스크린샷/첨부 파일은 v1에서 제외).
  - 시스템 프롬프트:
    - 의료/개인정보 가능성을 명시.
    - 주민번호/전화번호/환자명 등 PII 원문을 요구하지 말고 “마스킹된 키값”이나 “내부 환자 ID” 형태로 요청하도록 가이드.
  - 모델 출력(JSON):
    - `missing_fields`: 누락 가능성이 있는 필드 이름 리스트.
    - `questions_to_ask`: 고객/내부에 추가로 물어볼 질문 리스트.
    - `redmine_subject`: 추천 이슈 제목.
    - `redmine_description`: 정제된 Description 전문.
    - `confidence`: 0.0 ~ 1.0.
    - `risk_flags`: `["PII_POSSIBLE", ...]` 등 리스크 플래그.

- **출력(UI)**
  - JSON 원본 디버그 뷰.
  - 추천 제목/Description을 각각 편집 가능한 필드로 보여줌.
  - 질문 리스트를 보기 좋게 표시 (복사 버튼 제공).

#### 5.3. 고객사 → Repo/Branch 매핑 & 코드 후보 추천 (basic mapping)

- **고객사 레지스트리 파일**
  - 위치: `ops-issue-assistant/customers.json`
  - 포맷 예시:
    - `id`, `name`.
    - `source_targets`: 배열.
      - `label`: UI에 표시될 이름 (예: `"A병원 main (customer/a-hospital)"`).
      - `repo_url`: 참고용(현재는 정보성, 실제 git 연동은 v1에서 강제하지 않음).
      - `branch`: 기본 분석 기준 브랜치 이름(정보성).
      - `local_path`: 개발자 PC/서버에서 **이미 클론되어 있는 로컬 경로**.
      - `paths`: 검색 범위를 제한할 디렉터리 리스트 (예: `["front-end/", "back-end/"]`).
      - `priority`: 다수 타겟일 때 기본 선택 우선 순위.

- **검색 입력**
  - 키워드1: 메뉴명/화면명/한글 텍스트.
  - 키워드2: API 경로(`/biz/ord/...`) 등.
  - 키워드3: 에러 코드/메시지 일부 등.

- **검색 처리 (basic)**
  - `local_path` 기준으로 `paths`에 포함된 디렉터리들만 재귀 탐색.
  - 대상 확장자:
    - `.vue`, `.ts`, `.tsx`, `.js`, `.java`, `.xml`, `.yml`, `.yaml`, `.sql`, `.md`, `.properties`.
  - 무시 디렉터리: `node_modules`, `dist`, `build`, `target` 등.
  - 파일 크기 제한(예: 2MB 초과 파일 스킵).
  - 각 파일 내용에 대해 키워드 등장 횟수로 단순 스코어링, 상위 N개(예: 10~12개)를 후보로 노출.
  - 각 후보에는:
    - 상대 경로.
    - 스코어.
    - 키워드가 처음 등장하는 라인 일부(발췌) 표시.

- **출력(UI)**
  - 리스트형:
    - `score X` + 상대 경로(복사 가능).
    - 발췌 코드를 code 블록으로 한 줄 정도 보여줌.
  - 개발자는 이 경로를 IDE에서 열어 확인.

#### 5.4. 레드마인 연동 (읽기 전용, v1)

- **기능**
  - Redmine REST API를 통해 기본 메타데이터를 읽어온다.
    - Trackers 목록 (`/trackers.json`).
    - Issue Statuses 목록 (`/issue_statuses.json`).
  - UI에서 현재 선택된 Tracker/Status에 대응하는 ID를 보여주거나, 복사하여 이슈 생성 시 참고 가능.
- **설정**
  - `.env`:
    - `REDMINE_BASE_URL`
    - `REDMINE_API_KEY`
    - `REDMINE_PROJECT_ID` (미래 쓰기 기능을 대비해 미리 정의 가능)
  - v1에서는 **실제 이슈 생성/수정은 하지 않고**, 추후 확장을 위해 함수 구조만 갖춰둔다.

### 6. 비기능 요구사항

- **플랫폼/스택**
  - Python 3.x + Streamlit 기반 웹앱 (내부망에서 실행, URL로 접속).
  - Anthropic Claude API 사용 (텍스트만).
  - Redmine REST API는 HTTPS를 사용하는 것을 전제로 함.

- **보안/개인정보**
  - LLM으로 전송되는 데이터는 **텍스트만**이며, 스크린샷/파일은 v1에서 전송하지 않는다.
  - 프롬프트에서 PII/PHI를 직접 요구하지 말고, 마스킹된 키값이나 내부 ID만 요청하도록 명시.
  - 고객사별 repo/local\_path는 사내망/개발자 PC 상의 경로만 사용.

- **성능**
  - 키워드 기반 파일 검색은 한 고객사 기준 수천 개 정도의 파일에서 수 초 이내 응답을 목표.
  - LLM 호출은 5~10초 내 응답이 일반적이며, UI에서 로딩 상태를 명확히 표시.

### 7. UI/UX 요구사항

- **전체 톤**
  - 이름처럼 **귀엽고 한국어 위주의 친근한 UI**.
  - 메인 컬러는 레드마인에 맞춰 **레드 계열**을 기본으로 사용.
  - 예전 **버디버디 같은 느낌의 메인 페이지**:
    - 라운드 박스, 부드러운 색감, 작은 캐릭터/아이콘(이슈요정 느낌) 사용 가능.

- **페이지 구성 (단일 페이지 within Streamlit)**
  - 좌측: 입력 영역.
    - “이슈 작성” 카드: Tracker/Status/날짜/고객사/내용 등.
    - “코드 후보 검색” 카드: 키워드 입력/고객사/소스 타겟 선택.
  - 우측: 출력 영역.
    - 템플릿 Description 미리보기.
    - Claude 분석 결과(추천 제목/Description/질문 리스트).
    - Redmine 메타데이터 조회/표시.
    - 코드 후보 파일 리스트.

- **카피/문구 스타일**
  - 버튼 예시:
    - “버그버디에게 정리 부탁하기”
    - “추가로 뭐가 더 필요할지 물어보기”
    - “추천 코드 확인하기”
  - 오류 메시지도 딱딱하기보다는:
    - “버그버디 혼자 힘으로는 어려웠어요. 설정을 한 번만 더 확인해주세요.”

### 8. 엣지 케이스 및 예외 처리

- **LLM 호출 실패**
  - 네트워크 에러/토큰 오류/타임아웃 등:
    - 사용자에게 “지금은 버그버디가 쉬는 중이에요. 나중에 다시 시도해 주세요.” 정도의 메시지 + 원인 코드 표시.
    - LLM 없이도 기본 템플릿은 항상 생성 가능해야 함.

- **Redmine 설정 미완료**
  - `REDMINE_BASE_URL` 또는 `REDMINE_API_KEY`가 없을 경우:
    - 레드마인 관련 섹션에 “연동 설정이 아직 안 되어 있어요. 템플릿 복붙만 사용할 수 있어요.” 안내.

- **customers.json 누락/오류**
  - 파일이 없거나 파싱 실패:
    - 고객사 선택 드롭다운은 비워두고, 상단에 “customers.json을 만들어 주세요” 경고.
  - 특정 고객사에 source\_targets가 없을 경우:
    - 코드 후보 검색 버튼을 비활성화하고, 이유 안내.

- **local\_path가 실제로 존재하지 않음**
  - 해당 경로에 디렉터리가 없을 때:
    - “로컬 경로를 찾을 수 없어요. 이 경로에 레포를 클론했는지 확인해 주세요.” 메시지 + 검색 중단.

- **파일 검색 시 성능 문제**
  - 매우 큰 리포/파일이 있을 때 시간 초과 가능성:
    - 확장자/크기/디렉터리 필터로 1차 완화.
    - 향후 필요 시, 검색 시간 제한/취소 버튼 고려.

- **민감 정보가 입력 필드에 포함된 경우**
  - 이슈 내용/로그에 주민번호/전화번호 등 패턴이 보일 가능성:
    - v1에서는 자동 마스킹을 **필수는 아님**. 다만:
      - 프롬프트 및 UI 메시지에서 “환자명/주민번호 등은 올리지 말아 주세요. 내부 ID로만 적어 주세요.” 안내.

### 9. 트레이드오프

- **텍스트만 LLM 전송 vs 스크린샷/로그 전송**
  - 장점:
    - 개인정보/보안 리스크를 크게 줄일 수 있음.
    - 초기 도입 시 보안/컴플라이언스 검토가 수월.
  - 단점:
    - 스크린샷 기반 화면 자동 판별/세밀한 로그 해석은 제한적.
  - v1 결정:
    - **텍스트만 전송**, 스크린샷은 사람이 참고하는 용도에 머무르게 설계.

- **Redmine read-only vs write 연동**
  - read-only:
    - 안전하고, 기존 레드마인 프로세스를 덜 건드림.
    - 적용/롤백이 쉽다.
  - write 연동:
    - 완전 자동화 가능하지만, 권한/프로젝트/워크플로우 정책을 모두 맞춰야 함.
  - v1 결정:
    - **read-only 메타데이터 조회까지만**, 이슈 생성/수정은 후속 버전에서.

- **간단 키워드 검색 vs 고급 코드 인덱싱/시맨틱 검색**
  - 키워드 검색:
    - 구현이 단순, 3시간 내 가능.
    - 정확도는 제한적이나 “후보 좁히기”에는 충분.
  - 시맨틱 검색:
    - 정확도/편의성 훌륭하나, 인덱싱/스토리지/업데이트 전략이 필요.
  - v1 결정:
    - **키워드 기반 단순 스코어링**으로 시작, 추후 필요 시 시맨틱 검색 추가.
